# KLP-105 — ジェネリクスと特性境界

> 種別: Kotoba Language Proposal (KLP) — Standards Track（言語機能追加）  
> 番号: KLP-105  
> 題名: ジェネリクスと特性境界 — ユーザー定義型への型引数と能力制約  
> 状態: Draft  
> 注記: 本文書は言語仕様（`docs/仕様/`）への提案である。`[規範]` ラベルは含まず、
> 採択後に対応する仕様ファイルへ規範として移記される。

---

## 要約

`組`・`列挙`・`手順` の定義に**型引数**（型パラメータ）を付けられるようにする。
あわせて `手順` の型引数に**特性境界**（`〜を持つ`）を指定できるようにする。

既存の `一覧<T>` や `対応表<K,V>` と同等の再利用性をユーザー定義型でも実現できる。

---

## 動機

v1 時点でも `一覧<数>` や `対応表<文字列、数>` は使える。
しかしユーザーが `組` を定義するとき型引数を持てない。
これは「汎用データ構造が書けない」「KLP-104 の `選択肢<T>` が書けない」ことを意味する。

ジェネリクスが入ることで：

- `選択肢<T>`・`結果<T,E>`（KLP-104 の標準列挙型）が正式に書ける
- 「型を受け取る手順」が書け、重複コードが消える
- Rust の trait bounds 相当で「この型引数は特性 X を実装している必要がある」が言える

## 構文

### 2.1 EBNF 差分

```ebnf
(* 型引数リスト（定義側）*)
type_param_list  ::= "<" type_param_decl ("、" type_param_decl)* ">" ;
type_param_decl  ::= identifier ("は" identifier "を" "持つ")* ;

(* 型引数付き組・列挙・手順定義 *)
struct_def    ::= identifier type_param_list? "という" "組" struct_body ;
enum_def      ::= identifier type_param_list? "という" "列挙" enum_body ;
proc_def      ::= identifier type_param_list? "という" "手順" param_decl? ("→" type_expr)? block ;

(* type_expr に型引数適用を追加（既存 named_type の拡張）*)
named_type    ::= identifier ("<" type_expr ("、" type_expr)* ">")? ;
```

### 2.2 特性境界の構文

型引数宣言に境界を指定することで、「その型引数は指定した特性を実装していなければならない」という制約を付ける。

```text
<T は 挨拶できるを 持つ>
<T、E は エラーを 持つ>
```

## 意味論

### 3.1 型引数付き `組` の定義

```kotoba
包む<T> という 組
  中身は T

数値入りは 包む<数>を 作る 【中身: 42】
文字列入りは 包む<文字列>を 作る 【中身: 「こんにちは」】
```

型引数 `T` は定義時には具体型に束縛されない。
インスタンス生成時（または推論によって）具体型に単相化（monomorphize）される。

### 3.2 特性境界付き `手順`

```kotoba
挨拶させる<T は 挨拶できるを 持つ> という 手順 【誰:を〔T〕】
  誰の 挨拶するを 表示する
```

`T は 挨拶できるを 持つ` は「型引数 `T` が特性 `挨拶できる` を実装していなければならない」という制約。
境界を満たさない型を渡した場合はコンパイルエラー（GEN-103）。

### 3.3 単相化

ことば v1 は**単相化（monomorphization）**戦略を採用する。
型引数が具体型に決まるたびにその組み合わせのコードが生成される。
動的ディスパッチを使わないため、実行コストは具体型と等しい。

### 3.4 型検査規則

| 規範 ID | 内容                                                                         |
| ------- | ---------------------------------------------------------------------------- |
| GEN-101 | 型引数を持つ定義の利用時、すべての型引数が推論または明示されなければならない |
| GEN-102 | 型引数に適用する型は特性境界を満たさなければならない                         |
| GEN-103 | 境界を満たさない型を渡した場合、「`X` は `Y` を持っていません」エラーを報告  |
| GEN-104 | 型引数は定義ブロック内でのみ有効なスコープを持つ                             |
| GEN-105 | 型引数同士の等価性は構造的（名前ではなく制約で判定）としない。名前で判定する |

### 3.5 診断要件

| 診断コード | トリガー     | メッセージ例（日本語）                                                        |
| ---------- | ------------ | ----------------------------------------------------------------------------- |
| DGN-050    | GEN-101 違反 | 「`包む` の型引数 `T` が推論できません。`包む<数>` のように明示してください」 |
| DGN-051    | GEN-102/103  | 「`方向` は `挨拶できる` を持っていないため `挨拶させる` に渡せません」       |

## コード例

### 4.1 汎用スタック

```kotoba
スタック<T> という 組
  要素一覧は 一覧<T>

数スタックは スタック<数>を 作る 【要素一覧: 【】】
```

### 4.2 特性境界と組み合わせ

```kotoba
表示できる という 特性
  文字列化する という 手順 → 文字列

全部表示する<T は 表示できるを 持つ> という 手順 【一覧:を〔一覧<T>〕】
  一覧の 各要素で 繰り返す
    要素の 文字列化するを 表示する

点 という 組
  xは 数
  yは 数

点 は 表示できるを 持つ
  文字列化する という 手順
    「(」と それのX の x と 「, 」と それのX の y と 「)」を 結合する

点一覧は 【点を 作る 【x: 1、y: 2】、点を 作る 【x: 3、y: 4】】
全部表示するを 点一覧で 呼ぶ
```

### 4.3 KLP-104 の標準列挙型との組み合わせ

```kotoba
選択肢<T> という 列挙
  あり の【値は T】
  なし

安全に割る という 手順 【左:を〔数〕、右:を〔数〕】 → 選択肢<数>
  もし 右 が 0 と 等しい ならば
    選択肢のなし
  そうでなければ
    選択肢のあり【値: 左を右で割る】
```
