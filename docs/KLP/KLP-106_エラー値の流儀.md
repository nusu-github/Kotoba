# KLP-106 — エラー値の流儀

> 種別: Kotoba Language Proposal (KLP) — Standards Track（言語機能追加）  
> 番号: KLP-106  
> 題名: エラー値の流儀 — `結果` 型による値ベースエラーと `訴える` の使い分け  
> 状態: Draft  
> 注記: 本文書は言語仕様（`docs/仕様/`）への提案である。`[規範]` ラベルは含まず、
> 採択後に対応する仕様ファイルへ規範として移記される。

---

## 要約

ことば v1 には `試す`・`失敗した場合`・`訴える` による例外機構がある（EXC-001〜006）。
一方 KLP-104 では `結果<T,E>` 列挙型が導入される。
本提案は「例外と結果型をいつ使い分けるか」の**流儀（convention）**を規範として確定させる。

また、`結果` 型を扱う上で頻出するパターン（早期リターン・チェイン）の糖衣構文を追加する。

---

## 動機

例外と値ベースエラーが両方あると、書き手が「どちらを使えばいいのか」迷う。
Rust はパニック（回復不能）と `Result`（回復可能）を明確に分けており、コードが読みやすい。
ことばも同様の線引きを文書化することで設計が締まる。

あわせて、`結果<T,E>` をよく扱う状況で毎回 `どれかを調べる` を書くのは冗長である。
「失敗なら即 `返す`」パターンの糖衣（`?` 演算子相当）を導入する。

---

## 使い分け規則

### 2.1 `訴える`（例外）を使う場面

| 場面                         | 例                                   |
| ---------------------------- | ------------------------------------ |
| 内部の不変条件が壊れた       | 配列の範囲外アクセス、ゼロ除算       |
| 回復を意図しない致命的な状態 | 必要なファイルが存在しない（起動時） |
| プログラムのバグを報告する   | 型が来るはずのない場所に別の型が来た |

### 2.2 `結果<T,E>` を返す場面

| 場面                       | 例                                           |
| -------------------------- | -------------------------------------------- |
| 呼び出し側が回復できる失敗 | ファイルが見つからない（パスを変えて再試行） |
| ライブラリの API の境界    | 外部から呼ばれる公開手順のほぼすべて         |
| 失敗が正常なビジネスフロー | ログイン失敗・バリデーションエラー           |

## 構文（糖衣）

### 3.1 `問う` 演算子（早期リターン糖衣）

`結果<T,E>` または `選択肢<T>` を返す式の後ろに `を 問う` を付けると、
失敗時に現在の手順から失敗値を即座に返す。

```ebnf
(* postfix_expr に追加 *)
postfix_expr  ::= ... | te_step_stop "を" "問う" ;
```

#### 意味論

```text
式を 問う
```

は以下の脱糖に等しい：

```kotoba
どれかを調べる 式
  結果の成功【値: v】 ならば  v
  結果の失敗【理由: e】 ならば  結果の失敗【理由: e】を 返す
```

`選択肢<T>` の場合：

```kotoba
どれかを調べる 式
  選択肢のあり【値: v】 ならば  v
  選択肢のなし ならば  選択肢のなしを 返す
```

### 3.2 予約語の追加

| 区分 | 追加する予約語 |
| ---- | -------------- |
| 演算 | `問う`         |

---

## 意味論の規則

| 規範 ID | 内容                                                                                     |
| ------- | ---------------------------------------------------------------------------------------- |
| ERR-001 | `訴える` は回復不能な内部エラーにのみ使う（SHOULD。強制ではないが規約とする）            |
| ERR-002 | 公開手順のエラー返却は `結果<T,E>` を使う（SHOULD）                                      |
| ERR-003 | `を 問う` は `結果<T,E>` または `選択肢<T>` 型の式にのみ適用できる                       |
| ERR-004 | `を 問う` を使う手順の戻り型は `結果<T,E>` または `選択肢<T>` でなければコンパイルエラー |
| ERR-005 | `を 問う` の失敗時に返す値の型 `E` は現在の手順の戻り型の失敗型と一致しなければならない  |

### 診断要件

| 診断コード | トリガー     | メッセージ例（日本語）                                                          |
| ---------- | ------------ | ------------------------------------------------------------------------------- |
| DGN-060    | ERR-003 違反 | 「`問う` は `結果` か `選択肢` 型にのみ使えます」                               |
| DGN-061    | ERR-004 違反 | 「`問う` を使うには手順の戻り型を `結果` または `選択肢` にしてください」       |
| DGN-062    | ERR-005 違反 | 「失敗型が一致しません。`結果<数,文字列>` の手順に `結果<数,数>` は渡せません」 |

## コード例

### 4.1 `問う` による早期リターンチェイン

```kotoba
ファイル処理する という 手順 【パス:を〔文字列〕】 → 結果<文字列、文字列>
  内容は ファイルを パスで 読むを 問う     ※ 失敗なら即リターン
  変換済みは 内容を 変換するを 問う        ※ 失敗なら即リターン
  結果の成功【値: 変換済み】
```

### 4.2 `問う` なし版（比較用）

```kotoba
ファイル処理する という 手順 【パス:を〔文字列〕】 → 結果<文字列、文字列>
  どれかを調べる ファイルを パスで 読む
    結果の成功【値: 内容】 ならば
      どれかを調べる 内容を 変換する
        結果の成功【値: 変換済み】 ならば
          結果の成功【値: 変換済み】
        結果の失敗【理由: e】 ならば
          結果の失敗【理由: e】を 返す
    結果の失敗【理由: e】 ならば
      結果の失敗【理由: e】を 返す
```

ネストが深くなることが分かる。`問う` がこの冗長さを解消する。

### 4.3 `訴える` の正当な使用

```kotoba
除算する という 手順 【左:を〔数〕、右:を〔数〕】 → 数
  もし 右 が 0 と 等しい ならば
    「内部エラー: ゼロ除算は呼び出し前に検査すること」を 訴える
  左を 右で 割る
```
