# KLP-101 — `組` と `特性` の意味論

> 種別: Kotoba Language Proposal (KLP) — Standards Track（言語機能追加）  
> 番号: KLP-101  
> 題名: `組` と `特性` — データモデルの意味論と型規則  
> 状態: Draft  
> 注記: 本文書は言語仕様（`docs/仕様/`）への提案である。`[規範]` ラベルは含まず、
>       採択後に対応する仕様ファイルへ規範として移記される。

---

## 要約

`組`（構造体）と `特性`（トレイト）について、構文としての枠組みはすでに v1 に存在する。
しかし、フィールド定義の型・初期化・フィールド参照（`XのY`）・メソッド解決・特性境界といった
**意味論と型規則が未確定**のため、実際にデータをモデリングする用途で使えない。

本提案は「データモデル」層に相当するこれらの意味論を規範として確定させる。

---

## 動機

ことば v1 には `struct_def`・`trait_def`・`trait_impl` の構文定義がある。しかし：

- `組` のフィールドをどう宣言するか・初期化するかが未定義
- `組のインスタンス.フィールド` アクセスが `の` アクセスの一般規則に含まれるかどうかが曖昧
- `特性` のメソッドが静的ディスパッチか動的ディスパッチかが未定義
- `「Xは Yを 持つ」` で実装が揃っているかどうかをどう検査するかが未定義

この空白を放置すると「`組` と `特性` を書いても型検査されない雰囲気インターフェース」になる。
Python の「データを作って回す」体験と同等のことがことばでできるようにするには、
この層を先に固める必要がある。

---

## 構文

### 2.1 EBNF 差分

既存の `struct_def`・`trait_def`・`trait_impl` を拡張する。

```ebnf
(* 既存 struct_def を拡張 *)
struct_def    ::= identifier "という" "組" struct_body ;
struct_body   ::= INDENT field_decl+ DEDENT ;
field_decl    ::= identifier "は" type_expr sep ;

(* 既存 trait_def を拡張 *)
trait_def     ::= identifier "という" "特性" trait_body ;
trait_body    ::= INDENT method_sig+ DEDENT ;
method_sig    ::= identifier "という" "手順" param_decl ("→" type_expr)? sep ;

(* 既存 trait_impl を拡張 *)
trait_impl    ::= identifier "は" identifier "を" "持つ" impl_body ;
impl_body     ::= INDENT method_def+ DEDENT ;
method_def    ::= proc_def ;

(* 型式（型注釈で共通利用）*)
type_expr     ::= base_type | list_type | map_type | proc_type | named_type ;
base_type     ::= "数" | "文字列" | "真偽" | "無" ;
list_type     ::= "一覧" "<" type_expr ">" ;
map_type      ::= "対応表" "<" type_expr "、" type_expr ">" ;
proc_type     ::= "手順" param_type_decl ("→" type_expr)? ;
param_type_decl ::= "【" type_param ("、" type_param)* "】" ;
type_param    ::= type_expr ":" role_particle ;
named_type    ::= identifier ("<" type_expr ("、" type_expr)* ">")? ;

(* 組リテラル（初期化式）*)
primary_expr  ::= ... | struct_lit ;
struct_lit    ::= identifier "を" "作る" "【" field_init ("、" field_init)* "】" ;
field_init    ::= identifier ":" expr ;
```

### 2.2 予約語の追加

| 区分   | 追加する予約語 |
| ------ | -------------- |
| 型注釈 | `→`            |

`→` は ASCII `->` の日本語代替ではなく単独の記号トークンとして定義する。
字句規則において `→` は最長一致により `>` や `-` に先行して認識される。

## 意味論

### 3.1 `組` の意味論

**フィールド定義**

`組` ブロック内の各 `field_decl` はフィールド名と型を宣言する。
フィールドの順序は宣言順で保持される。

```kotoba
点 という 組
  xは 数
  yは 数
```

**初期化**

`組を作る` 式で初期化する。フィールドはすべて明示指定しなければならない。
フィールドの記述順序は問わない（助詞構文と同様、名前で解決する）。

```kotoba
p は 点を 作る 【x: 3、y: 4】
```

**フィールドアクセス**

`XのY` アクセス規則（`access_expr`）を `組` インスタンスのフィールド参照に拡張する。
`p の x` は `p` が `点` 型であるとき、`数` 型の値を返す。

**再束縛と可変性**

フィールドの可変性は束縛変数の可変性に従う。

- `p は 点を 作る 【…】` — `p` は不変束縛。フィールドへの再代入禁止。
- `変わる q は 点を 作る 【…】` — `q` は可変束縛。`q の x を 5に 変える` が許可される。

### 3.2 `特性` の意味論

**メソッドシグネチャ**

`特性` ブロック内の各 `method_sig` はメソッド名・引数の助詞・戻り型を宣言する。
本体は含まない（インターフェース定義のみ）。

```kotoba
形 という 特性
  面積を計算する という 手順 → 数
  周長を計算する という 手順 → 数
```

**実装（`持つ`）**

`Xは Yを 持つ` ブロックで `X` が特性 `Y` を実装する。
ブロック内にはシグネチャが要求する全メソッドを定義しなければならない。

```kotoba
円 という 組
  半径は 数

円 は 形を 持つ
  面積を計算する という 手順
    それのX の 半径 と 3.14159の積
  周長を計算する という 手順
    それのX の 半径 と 2の積 と 3.14159の積
```

**ディスパッチ方式**

ことば v1 は**静的ディスパッチ**を採用する。
コンパイル時に型が確定している場合、メソッド呼び出しは直接解決される。
動的ディスパッチ（型消去・vtable）は将来の提案に委ねる。

### 3.3 型検査規則

| 規則    | 内容                                                                      |
| ------- | ------------------------------------------------------------------------- |
| STR-001 | `組を作る` 式はすべてのフィールドが指定されていなければコンパイルエラー   |
| STR-002 | フィールド値の型は宣言された型と一致しなければならない                    |
| STR-003 | `XのY` アクセスにおいて `Y` が `X` のフィールドでなければコンパイルエラー |
| STR-004 | `Xは Yを 持つ` において宣言されたメソッドが欠けていればコンパイルエラー   |
| STR-005 | メソッドのシグネチャ（助詞・戻り型）が特性の宣言と一致しなければエラー    |
| STR-006 | 不変束縛の `組` インスタンスのフィールドへ再代入を試みた場合エラー        |

### 3.4 診断要件

| 診断コード | トリガー     | メッセージ例（日本語）                                                    |
| ---------- | ------------ | ------------------------------------------------------------------------- |
| DGN-010    | STR-001 違反 | 「点を作るには `y` フィールドの指定が必要です」                           |
| DGN-011    | STR-002 違反 | 「`x` には `数` が必要ですが `文字列` が渡されました」                    |
| DGN-012    | STR-003 違反 | 「`点` には `z` フィールドがありません」                                  |
| DGN-013    | STR-004 違反 | 「`円` は `形` を実装していますが `周長を計算する` が定義されていません」 |
| DGN-014    | STR-005 違反 | 「`面積を計算する` の戻り型が `形` の要求する `数` と一致しません」       |
| DGN-015    | STR-006 違反 | 「`p` は不変束縛のため `x` を変えることができません」                     |

## コード例

### 4.1 基本的な組とフィールドアクセス

```kotoba
点 という 組
  xは 数
  yは 数

p は 点を 作る 【x: 3、y: 4】
p の x を 表示する    ※ → 3
```

### 4.2 特性の定義と実装

```kotoba
挨拶できる という 特性
  挨拶する という 手順 → 文字列

人 という 組
  名前は 文字列

人 は 挨拶できるを 持つ
  挨拶する という 手順
    「こんにちは、」と それのX の 名前 の 結合

太郎 は 人を 作る 【名前: 「太郎」】
太郎 の 挨拶する を 表示する
```

### 4.3 可変フィールド

```kotoba
カウンター という 組
  値は 数

変わる c は カウンターを 作る 【値: 0】
c の 値を c の 値 と 1 の 和に 変える
c の 値を 表示する    ※ → 1
```

## 非互換性影響分析

本節では、KLP-101 採択時の非互換を次の基準で数える。比較対象は v1 正式仕様
（`docs/仕様/02_構文.md`、`docs/仕様/03_名前束縛とスコープ.md`、`docs/仕様/04_呼び出しとて形.md`）とする。

- v1 で受理されていたソースが構文として受理されなくなる（パース不能、または別構文へ吸収される）
- 同じトークン列でも解釈カテゴリや解決規則が変わる
- 既存 MUST / MUST NOT 規範と衝突する

### A. 構文レベルの非互換（高優先）

v1 では `struct_def`・`trait_def`・`trait_impl` の本体はいずれも `block` であり、
`statement` を受理できる（SYN-001、`docs/仕様/02_構文.md`）。一方 KLP-101 は本体を
`field_decl+` / `method_sig+` / `method_def+` に限定しているため、受理範囲が縮小する。

`v1 で受理 -> KLP-101 採択後に非受理` となる代表例：

- `組` ブロック内で、フィールド宣言以外の文（束縛文、式文、補助手順定義）を置く
- `特性` ブロック内で、シグネチャではなく `proc_def`（本体付き手順）を置く
- `X は Yを 持つ` ブロック内で、`proc_def` 以外（束縛文や式文）を置く

加えて `組` 内の `xは 数` は、v1 では `bind_stmt` に分類されるトークン列だが、
KLP-101 では `field_decl` に分類される。これは同一トークン列のカテゴリ転換であり、
意味変更（束縛からフィールド宣言への変換）に該当する。

### B. 既存規範との衝突

フィールド代入形 `q の x を 5に 変える` は、v1 の
`rebind_stmt ::= identifier "を" expr "に" "変える"` と整合しない。
また規範上も、SCP-002 は再束縛を `名前を 式に 変える` のみに限定しているため、
アクセス式左辺の再束縛は現行規範と衝突する。

メソッド呼び出しも、受け手（レシーバ）受け渡しをどう規定するかにより、
CAL-005（メソッド呼び出しにおける暗黙 `が` 注入の禁止）との衝突リスクが高い。
KLP-101 例の `太郎 の 挨拶する` は見かけ上 `XのY` 形であり、呼び出し時に暗黙補完を導入すると
CAL-005 の MUST NOT に抵触し得る。

### C. 字句・予約語の互換

KLP-101 が導入する `→` と型式の `<` `>` は、v1 で明示済みの字句記号集合にないため、
字句仕様差分を発生させる（`docs/仕様/01_字句.md` との不一致解消が必要になる）。
これは主に「従来字句エラーだった入力の受理拡大」に寄るが、識別子文字集合を広く扱う実装では、
実装依存で既存コードを壊す余地がある。

また `数` / `文字列` / `一覧` / `対応表` / `手順` を予約語へ昇格させる設計を採る場合、
それらを束縛名に使っていた既存コードは失敗する。v1 は SCP-008 により
「予約語を束縛名として使用してはならない」を規範化しているため、予約語集合の変更は
直接的な束縛互換破壊になる。

### D. 適合性運用上の必須追従

KLP-101 を規範採択する場合、非互換差分は DIFF-001 に従って理由付きで列挙される必要がある。
また、KLP-101 が新設する規則群（STR-001〜006）については、CNF-002 により
受理/拒否テストの対応が必要である。さらに、規範本文で参照する構文は CNF-005 に従い
EBNF 上で定義されていなければならない。

これらは実装手段の提示ではなく、採択時に満たすべき適合要件としての影響である。

### E. 非互換サマリ（重要度）

| 論点                              | 非互換の種類                       | 重要度 | 破壊・衝突内容                                          |
| --------------------------------- | ---------------------------------- | ------ | ------------------------------------------------------- |
| `組/特性/持つ` 本体制限           | 既存 v1 受理コードの非受理化       | 高     | v1 は `block` 受理、KLP-101 は専用本体のみ受理          |
| `組` 内 `xは 数` の意味転換       | 同一トークン列の意味変更           | 高     | `bind_stmt` と `field_decl` のカテゴリ転換              |
| フィールド代入と `SCP-002` 衝突   | 既存規範/EBNF との衝突             | 高     | 再束縛左辺が識別子限定という v1 制約と不整合            |
| メソッド呼び出しと `CAL-005` 衝突 | 既存 MUST NOT との衝突リスク       | 高     | 暗黙 `が` 注入に依存する解決規則は CAL-005 に抵触し得る |
| `→` / `< >` の字句差分            | 字句仕様差分（実装依存破壊を含む） | 中     | 記号集合拡張に伴う処理系差分露出                        |
| 型名予約語化による束縛破壊        | 予約語集合変更による互換破壊       | 中     | SCP-008 により既存の同名束縛が不許可化                  |

結論として、最大の非互換要因は `block` から専用本体への構文縮退である。
加えて、再束縛規則（SCP-002）と呼び出し規則（CAL-005）との整合欠落が
二次的だが重大な非互換リスクを形成する。
