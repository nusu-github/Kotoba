## 要約

言語機能の充実とは独立に、**開発者体験（DX）**を向上させる 3 つのツールを定義する：

1. **REPL**（対話的実行環境）
2. **デバッガ**（ステップ実行・変数検査）
3. **フォーマッタ**（コードの自動整形）

言語機能の追加より「現代的な言語を触っている」感触に直結するのはツール層であることが多い。
この 3 つが揃うと開発サイクルが一気に快適になる。

---

## 動機

「Python っぽく日常作業できる言語」の体感値は、構文だけでなくツールに大きく依存する。

- REPL がないと「一行試すためにファイルを作って実行する」コストが毎回かかる
- フォーマッタがないと「コードスタイルの議論」が発生してコードレビューが詰まる
- デバッガがないと `表示するを 呼ぶ` デバッグしかできない

これらは言語機能より実装コストが高いこともあるが、
仕様として要件を先に固めることで実装がブレなくなる。

## 1. REPL（対話的実行環境）

### 仕様要件

`kotoba` コマンドを引数なしで起動した場合、REPL モードで動作する。

| 要件    | 内容                                                                       |
| ------- | -------------------------------------------------------------------------- |
| TOOL-001 | `kotoba` を引数なしで実行すると REPL が起動する                           |
| TOOL-002 | 各行の入力を評価し、結果の値を `→ 値` 形式で表示する                      |
| TOOL-003 | 式が `無` を返す場合は出力しない                                           |
| TOOL-004 | 前の入力の束縛はセッション内で保持される（セッションスコープ）             |
| TOOL-005 | `インデント` による複数行入力をサポートする（ブロックが閉じるまで継続）    |
| TOOL-006 | `:q` または Ctrl-D で REPL を終了する                                      |
| TOOL-007 | 評価時の例外はセッションを終了させずに診断を表示して継続する               |

### REPL 表示例

```
言（ことば）v1.0 対話モード。終了するには :q を入力してください。

>>> x は 42
>>> x と 1 の 和
→ 43
>>> もし x が 40 より 大きい ならば
...   「大きい」
... そうでなければ
...   「小さい」
→ 「大きい」
>>> 0 を 0 で 割る
! エラー: ゼロ除算（09行目）
>>> :q
```

## 2. デバッガ

### 仕様要件

`kotoba debug ファイル.kb` で対話的デバッガが起動する。

| 要件     | 内容                                                                        |
| -------- | --------------------------------------------------------------------------- |
| TOOL-010 | ブレークポイントをファイル名と行番号で設定できる                            |
| TOOL-011 | `次へ`（step over）・`入る`（step into）・`続ける`（continue）コマンドを提供する |
| TOOL-012 | 現在のスコープの全束縛変数と値を一覧表示できる                              |
| TOOL-013 | 任意の式をその場で評価できる（ウォッチ式）                                  |
| TOOL-014 | スタックトレースを表示できる                                                |
| TOOL-015 | `ここ` / `そこ` 語彙（既存の診断情報指示語）とデバッガを統合する方針を検討する |

### デバッガコマンド一覧（参考）

| コマンド       | 説明                         |
| -------------- | ---------------------------- |
| `次へ`         | 1 行進む（step over）        |
| `入る`         | 手順呼び出しに入る（step into）|
| `出る`         | 現在の手順から出る（step out） |
| `続ける`       | 次のブレークポイントまで実行 |
| `止める N行目` | N 行目にブレークポイントを設定 |
| `調べる 式`    | 式を評価して表示             |
| `一覧`         | 現在のスコープの変数一覧     |

## 3. フォーマッタ

### 仕様要件

`kotoba 整形する ファイル.kb` でコードを自動整形する。

| 要件     | 内容                                                                        |
| -------- | --------------------------------------------------------------------------- |
| TOOL-020 | フォーマッタは冪等である（整形後に再度整形しても変化がない）               |
| TOOL-021 | インデントは半角スペース 2 個を標準とする                                   |
| TOOL-022 | `。` と `NEWLINE` の使い分けを正規化する（どちらか一方に統一）             |
| TOOL-023 | コメント（`※`）の前後のスペースを正規化する                                |
| TOOL-024 | `kotoba 整形する --確認` で整形が必要かどうかだけを検査する（CI 用）       |
| TOOL-025 | フォーマッタはパーサと同じ文法を使い、構文エラーのあるファイルは拒否する   |

### 整形規則（主要なもの）

| 項目                     | 規則                                             |
| ------------------------ | ------------------------------------------------ |
| インデント               | 半角スペース 2 個（タブ禁止）                    |
| 文区切り                 | `NEWLINE` に統一（`。` を自動変換）              |
| 演算子の前後             | スペースを 1 個ずつ挿入                          |
| `【` `】` の内側         | スペースなし                                     |
| `｛` `｝` の内側         | スペースなし                                     |
| 連続する空行             | 最大 1 行に縮小                                  |
| `※` コメントの前後       | `※` の後ろにスペース 1 個                        |
