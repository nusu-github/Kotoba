# KLP-104 — `列挙` 型（sum type）

> 種別: Kotoba Language Proposal (KLP) — Standards Track（言語機能追加）  
> 番号: KLP-104  
> 題名: `列挙` 型 — 直和型（sum type）の導入  
> 状態: Draft  
> 注記: 本文書は言語仕様（`docs/仕様/`）への提案である。`[規範]` ラベルは含まず、
>       採択後に対応する仕様ファイルへ規範として移記される。

---

## 要約

`列挙` キーワードを導入し、複数の値の形（バリアント）を一つの型として束ねる**直和型**を追加する。
Rust の `enum`・Haskell の `data` 相当。既存の `どれかを調べる`（match）と組み合わせることで、
`Option<T>` や `Result<T,E>` 相当を標準ライブラリなしに自前定義できるようになる。

---

## 動機

ことば v1 には `組`（積型）があるが、「AかBかCのどれか」を表す**直和型**がない。
直和型がないと：

- エラーが起きるかもしれない処理の結果を型で表現できない
- 「値があるかもしれない」状況をヌルポインタ相当で扱うしかない
- `どれかを調べる` が威力を発揮する相手（データ型の分岐）がない

`列挙` が入ると `どれかを調べる` との噛み合わせが完成し、Rust 的な「型で状態を表す」設計が可能になる。

### `組` と `列挙` の違い

| 特徴           | `組`（積型）                      | `列挙`（直和型）                  |
| -------------- | --------------------------------- | --------------------------------- |
| 意味           | 複数フィールドを**すべて**持つ    | 複数バリアントの**どれか一つ**    |
| 対応するもの   | Rust struct、Python dataclass     | Rust enum、Haskell data           |
| `調べる` との相性 | フィールドアクセス              | パターンマッチの主な対象          |

## 構文

### 2.1 EBNF 差分

```ebnf
(* 文に enum_def を追加 *)
statement     ::= ... | enum_def ;
public_decl   ::= "公開" (proc_def | struct_def | trait_def | enum_def) ;

(* 列挙型定義 *)
enum_def      ::= identifier "という" "列挙" enum_body ;
enum_body     ::= INDENT variant_decl+ DEDENT ;
variant_decl  ::= identifier ("の" "【" field_decl+ "】")? sep ;

(* バリアント生成式 *)
primary_expr  ::= ... | variant_expr ;
variant_expr  ::= type_name "の" variant_name ("【" field_init ("、" field_init)* "】")? ;
type_name     ::= identifier ;
variant_name  ::= identifier ;

(* match アームのパターンを拡張 *)
match_arm     ::= pattern "ならば" block ;
pattern       ::= literal
               | identifier                           (* 束縛パターン *)
               | "_"                                  (* ワイルドカード *)
               | type_name "の" variant_name ("【" bind_pattern+ "】")? ; (* バリアントパターン *)
bind_pattern  ::= identifier ":" identifier ;          (* フィールド名: 束縛名 *)
```

### 2.2 予約語の追加

| 区分   | 追加する予約語 |
| ------ | -------------- |
| 定義   | `列挙`         |

## 意味論

### 3.1 バリアント定義

```kotoba
結果型 という 列挙
  成功の【値は 数】
  失敗の【理由は 文字列】
```

- `成功` と `失敗` が `結果型` の**バリアント**である。
- バリアントはフィールドを持てる（0 個以上）。
- フィールドなしのバリアントも許可する（列挙定数相当）。

```kotoba
方向 という 列挙
  北
  南
  東
  西
```

### 3.2 バリアント生成

```kotoba
成功値は 結果型の成功【値: 42】
失敗値は 結果型の失敗【理由: 「ゼロ除算」】
今の方向は 方向の北
```

### 3.3 `どれかを調べる` によるパターンマッチ

```kotoba
結果型 の バリアントを どれかを調べる
  結果型の成功【値: v】 ならば
    vを 表示する
  結果型の失敗【理由: msg】 ならば
    「エラー: 」と msgを 結合して 表示する
```

### 3.4 標準列挙型（組み込み定義）

以下の 2 つをプリルード（自動導入）として定義する：

```kotoba
公開 選択肢<T> という 列挙
  あり の【値は T】
  なし

公開 結果<T、E> という 列挙
  成功の【値は T】
  失敗の【理由は E】
```

これにより Rust の `Option<T>`・`Result<T,E>` に相当する安全なエラー処理が書ける。

### 3.5 型検査規則

| 規範 ID | 内容                                                                         |
| ------- | ---------------------------------------------------------------------------- |
| ENM-001 | `列挙` の各バリアント名は同一型内で一意でなければならない                   |
| ENM-002 | バリアント生成式において、すべてのフィールドを指定しなければならない         |
| ENM-003 | フィールド値の型は宣言された型と一致しなければならない                      |
| ENM-004 | `どれかを調べる` において全バリアントが網羅されない場合は警告（SHOULD）      |
| ENM-005 | `_`（ワイルドカード）パターンは常に一致し、束縛を生成しない                 |
| ENM-006 | バリアントパターンのフィールド束縛は `どれかを調べる` ブロック内でのみ有効  |

### 3.6 診断要件

| 診断コード | トリガー    | メッセージ例（日本語）                                           |
| ---------- | ----------- | ---------------------------------------------------------------- |
| DGN-040    | ENM-001 違反 | 「`結果型` にはすでに `成功` バリアントが定義されています」      |
| DGN-041    | ENM-002 違反 | 「`成功` バリアントには `値` フィールドの指定が必要です」        |
| DGN-042    | ENM-004      | 「`結果型の失敗` バリアントが `どれかを調べる` で扱われていません（ヒント: `_` で全体をキャッチできます）」 |

## コード例

### 4.1 エラーを値として扱う

```kotoba
ゼロ除算する という 手順 【左:を〔数〕、右:を〔数〕】 → 結果<数、文字列>
  もし 右 が 0 と 等しい ならば
    結果の失敗【理由: 「ゼロ除算」】
  そうでなければ
    結果の成功【値: 左を右で割る】

結果は ゼロ除算するを 10で 5で 呼ぶ
どれかを調べる 結果
  結果の成功【値: v】 ならば
    「答え: 」と vを 結合して 表示する
  結果の失敗【理由: msg】 ならば
    「失敗: 」と msgを 結合して 表示する
```

### 4.2 フィールドなしバリアント（列挙定数）

```kotoba
季節 という 列挙
  春
  夏
  秋
  冬

今の季節は 季節の夏
どれかを調べる 今の季節
  季節の春 ならば 「暖かい」を 表示する
  季節の夏 ならば 「暑い」を 表示する
  季節の秋 ならば 「涼しい」を 表示する
  季節の冬 ならば 「寒い」を 表示する
```
