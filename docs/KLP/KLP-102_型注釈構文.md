# KLP-102 — 型注釈構文

> 種別: Kotoba Language Proposal (KLP) — Standards Track（言語機能追加）  
> 番号: KLP-102  
> 題名: 型注釈構文 — 推論不能箇所のための最小注釈  
> 状態: Draft

## 要約

ことば v1 は静的型推論中心で動作し、推論不能な箇所はコンパイルエラーとする（TYP-002）。
しかし「推論不能」は現実にしばしば起きる。その箇所に書ける**最小の型注釈構文**を導入する。

対象は以下の 3 箇所に限定する：

1. 手順の仮引数型
2. 手順の戻り型
3. ローカル束縛の型

この 3 箇所に注釈を書ければ、型推論が機能しない状況のほぼ全域をカバーできる。

---

## 動機

v1 の方針「推論できなければエラー」は設計として正しい。しかし、現実のコードでは：

- 引数が複数の型を受け取り得る位置
- 再帰手順の戻り型（型推論器が循環を解けない場合）
- 空の一覧 `【】` を宣言してから後で要素を追加する場合

などで推論が止まる。注釈がなければ「書けないコード」が発生する。

Rust では `: T` で型を明示するように、ことばでも**読んで分かる自然な形**で型を書ける場所を用意する。
注釈は「書かなくてよいときは書かない」が基本。書いたときのみ検査が追加される。

## 構文

### 2.1 EBNF 差分

```ebnf
(* 仮引数宣言に型注釈を追加 *)
param         ::= identifier ":" role_particle ("〔" type_expr "〕")? ;

(* 手順定義に戻り型注釈を追加 *)
proc_def      ::= identifier "という" "手順" param_decl? ("→" type_expr)? block ;

(* 束縛文に型注釈を追加 *)
bind_stmt         ::= bind_target ("〔" type_expr "〕")? "は" expr ;
mutable_bind_stmt ::= "変わる" bind_target ("〔" type_expr "〕")? "は" expr ;

(* type_expr は KLP-101 で定義。再掲 *)
type_expr     ::= base_type | list_type | map_type | proc_type | named_type ;
```

### 2.2 記法の選択

`〔 〕`（隅付き括弧）を型注釈の括り記号として採用する理由：

| 選択肢   | 理由                                                                |
| -------- | ------------------------------------------------------------------- |
| `〔型〕` | 日本語文書での〔 〕は「補足・注釈」の語感があり、型注釈の意味と合う |
| `<型>`   | HTML/XML との混同、比較演算子との衝突                               |
| `（型）` | 日本語の括弧は呼び出し式の `( )` と衝突する可能性                   |
| `: 型`   | `:` は既存の助詞区切り（`param` 内）と重複する                      |

`〔 〕` は予約トークンとして追加し、識別子内には現れない。

## 意味論

### 3.1 仮引数型注釈

```kotoba
加算する という 手順 【左:を〔数〕、右:を〔数〕】 → 数
  左 と 右 の 和
```

- `〔数〕` は「この引数の型は `数` である」という制約を追加する。
- 注釈がない場合、型は呼び出し側の型から推論される（既存の TYP-001 の適用）。
- 注釈がある場合、呼び出し側の型が注釈と一致しなければコンパイルエラー（ANN-001）。

### 3.2 戻り型注釈

```kotoba
絶対値を求める という 手順 【値:を〔数〕】 → 数
  もし 値 が 0 より 大きい ならば
    値
  そうでなければ
    値と-1の積
```

- `→ 型` は「この手順の戻り値の型は指定した型でなければならない」という制約。
- 本体の型推論結果が `→` の型と一致しなければコンパイルエラー（ANN-002）。
- 戻り型注釈のない手順は本体から推論する（既存の挙動）。

### 3.3 束縛型注釈

```kotoba
空一覧〔一覧<数>〕は 【】
```

- `束縛名〔型〕は 式` の形で束縛時の型を固定できる。
- 空リテラル `【】` や `｛｝` のように型が確定しない式に対して特に有効。
- 注釈型と推論型が一致しなければコンパイルエラー（ANN-003）。

### 3.4 型検査規則

| 規範 ID | 内容                                                                     |
| ------- | ------------------------------------------------------------------------ |
| ANN-001 | 仮引数に型注釈がある場合、呼び出し側の型が注釈型と一致しなければエラー   |
| ANN-002 | 戻り型注釈がある場合、本体の推論型が注釈型と一致しなければエラー         |
| ANN-003 | 束縛に型注釈がある場合、右辺の推論型が注釈型と一致しなければエラー       |
| ANN-004 | 型注釈は必須ではない。注釈がない場合は通常の型推論を適用する             |
| ANN-005 | 型注釈と推論結果が両方存在し矛盾する場合、注釈を優先してエラーを報告する |

### 3.5 診断要件

| 診断コード | トリガー     | メッセージ例（日本語）                                                    |
| ---------- | ------------ | ------------------------------------------------------------------------- |
| DGN-020    | ANN-001 違反 | 「`左` には `数` が必要ですが `文字列` が渡されました」                   |
| DGN-021    | ANN-002 違反 | 「戻り型は `数` と宣言されていますが、本体の型は `文字列` です」          |
| DGN-022    | ANN-003 違反 | 「`空一覧` の型注釈は `一覧<数>` ですが、右辺の型は `一覧<文字列>` です」 |

## コード例

### 4.1 引数と戻り型の注釈

```kotoba
べき乗する という 手順 【底:を〔数〕、指数:を〔数〕】 → 数
  ※ 再帰手順。戻り型注釈がないと推論が循環する可能性がある
  もし 指数 が 0 と 等しい ならば
    1
  そうでなければ
    底 と こうを 底で 指数と-1の積で 呼ぶ の積
```

### 4.2 空コレクションの初期化

```kotoba
名前一覧〔一覧<文字列>〕は 【】
名前一覧 に 「太郎」を 追加する
名前一覧 に 「花子」を 追加する
```

### 4.3 注釈なし（推論が成功する場合）

```kotoba
※ 注釈不要。右辺から「数」と推論できる
カウンターは 0

※ 注釈不要。引数の使われ方から推論できる（単純な場合）
二倍する という 手順 【値:を】
  値 と 2 の 積
```
