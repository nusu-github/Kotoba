## 要約

ことばプログラムを**パッケージ**として定義・配布・依存解決するための仕様を導入する。
Rust の Cargo・Python の pip/pyproject.toml に相当する仕組みを、ことばの設計原則に合わせて定義する。

`v0.4 → v1` 差分文書（DIFF-001）では「保留候補」として先送りにされていたが、
言語が育つにつれ「パッケージ管理がない＝配布が詰まる」問題が顕在化するため、
早期に仕様の骨格を確立する。

---

## 動機

言語が「便利な道具を配布して共有する」ところまで成熟するには、
パッケージの定義・バージョン指定・依存解決の仕組みが必要になる。
ここがないと：

- 複数プロジェクト間でコードを再利用する手段がない
- バージョンが変わると互換性が壊れても追跡できない
- コミュニティが育つ前に「使い回しできない言語」という印象が定着する

## パッケージ定義ファイル

### 2.1 `使う場所.toml`（kotoba.toml）

パッケージのルートディレクトリに `使う場所.toml` を置く。
TOML 形式を採用する（Cargo.toml と同様のシンプルな設定フォーマット）。

```toml
[パッケージ]
名前 = "myapp"
バージョン = "1.0.0"
作者 = ["太郎 <taro@example.com>"]
説明 = "サンプルパッケージ"

[依存]
ことばの数学 = "2.1"
ことばのJSON = ">=1.0, <2.0"
```

### 2.2 フィールド定義

| フィールド名 | 必須 | 説明                                          |
| ------------ | ---- | --------------------------------------------- |
| `名前`       | YES  | パッケージ名（ASCII 英数字・ハイフン・アンダースコア）|
| `バージョン` | YES  | セマンティックバージョニング（`X.Y.Z` 形式）  |
| `作者`       | NO   | 作者の一覧（文字列の配列）                    |
| `説明`       | NO   | 一行の説明                                    |

### 2.3 バージョン指定書式

依存バージョンの書式（Python の pip・Cargo の要件をシンプル化）：

| 書式            | 意味                              |
| --------------- | --------------------------------- |
| `"1.2"`         | `>=1.2.0, <2.0.0`（互換バージョン）|
| `">=1.0, <2.0"` | 明示的な範囲指定                  |
| `"=1.0.0"`      | 完全固定                          |

## ディレクトリ構成

```
myapp/
├── 使う場所.toml        # パッケージ定義
├── src/
│   └── 主.kb            # エントリポイント（省略可能）
└── tests/
    └── テスト.kb
```

エントリポイントのデフォルトは `src/主.kb`。`使う場所.toml` の `[パッケージ]` セクションで変更可能。

## パッケージレジストリ

### 3.1 公式レジストリ

`kotoba.dev`（仮）を公式パッケージレジストリとする（将来）。
`kotoba add パッケージ名` でインストール・`使う場所.toml` に追記する CLI コマンドを提供する。

### 3.2 ローカル依存

```toml
[依存]
utils = { パス = "../utils" }
```

`パス` キーでローカルパスを指定できる（開発中のモノレポ等で使用）。

## バージョン解決規則

| 規範 ID | 内容                                                                         |
| ------- | ---------------------------------------------------------------------------- |
| PKG-001 | 依存関係は `使う場所.toml` で宣言し、バージョンはセマンティックバージョニングに従う |
| PKG-002 | 依存解決はコンパイル前に完了し、解決済みの正確なバージョン一覧を `依存.lock`（lockfile）に記録する |
| PKG-003 | `依存.lock` が存在する場合、それに従って解決し新たなバージョン解決は行わない  |
| PKG-004 | 同一パッケージの複数バージョンが依存グラフに現れた場合はエラーとする（ダイヤモンド問題を拒否）|
| PKG-005 | `公開` のない定義は外部パッケージから参照できない（MOD-003 と整合）           |

### 診断要件

| 診断コード | メッセージ例（日本語）                                                   |
| ---------- | ------------------------------------------------------------------------ |
| DGN-140    | 「パッケージ `ことばの数学` のバージョン `2.1` が見つかりません」        |
| DGN-141    | 「`ことばのJSON` のバージョン競合: `1.2` と `1.5` が要求されています。`>=1.5` に統一してください」 |

## CLI インターフェース（実装参考）

本節は規範でなく、実装者向けの参考である。

| コマンド                     | 動作                                   |
| ---------------------------- | -------------------------------------- |
| `kotoba 初期化する`          | 新しいパッケージを作成（`使う場所.toml` 生成） |
| `kotoba 追加する パッケージ名` | 依存を追加してインストール             |
| `kotoba 構築する`            | 現在のパッケージをビルド               |
| `kotoba 公開する`            | レジストリにパッケージを公開           |
