## 要約

`仮に〜ならば〜` を、通常の条件分岐（`もし〜ならば`）とは別の構文として導入する。  
`仮に` は **仮想実行式（ドライラン）** であり、「ある仮定を置いた世界で処理を走らせ、結果だけを受け取る」操作を表す。仮定ブロック・実行ブロック内の副作用は外側の環境に一切漏れない。例外は内部で捕捉され `そうでなければ` ブロックへ流れる。

---

## 動機

日本語の「もし〜」は *条件の真偽を判定する* 語感が強い。コードを読む側も「ここは分岐だ」と認識する。

一方「仮に〜」には異なるニュアンスがある。**仮定を置いた別世界線を一回走らせて、そこで何が起きるかを見たい**、という感覚だ。条件が真か偽かを確かめる前に、「こういう前提を置いたら、どんな結論・影響が生まれるか」を手に入れたい。

この違いをキーワードの違いだけで済ませてしまうのは勿体ない。`もし` と `仮に` が完全に同じ意味なら、どちらか一方で十分だ。`仮に` を **副作用を持たない仮想実行** として定義することで、二つのキーワードが本当に異なる操作を表すようになる。

### `もし` と `仮に` の違い

| 観点                    | `もし〜ならば`         | `仮に〜ならば`                       |
| ----------------------- | ---------------------- | ------------------------------------ |
| 意味モデル              | 条件が真なら分岐を実行 | 仮定世界を生成して実行               |
| 副作用                  | 外側の環境に反映される | **外側に漏れない**（ロールバック）   |
| 例外                    | 伝搬する               | **内部で捕捉**し `そうでなければ` へ |
| 戻り値                  | 選択された分岐の最終値 | `ならば` ブロックの最終値            |
| `そうでなければ` の役割 | else 分岐              | 失敗（例外）側の分岐                 |

## 構文

### 3.1 EBNF 差分

既存の `primary_expr` に `assume_expr` を追加する。

```ebnf
(* 既存の primary_expr に追加する項目 *)
primary_expr ::= ...
               | assume_expr ;       (* 追加 *)

assume_expr  ::= "仮に" block "ならば" block
                 ("そうでなければ" block)? ;
```

`assume_expr` の `block` は既存の `block` 規則（`INDENT statement+ DEDENT`）をそのまま使う。

### 3.2 予約語の追加

既存の LEX-010 予約語一覧（`docs/仕様/01_字句.md`）に `仮に` を追加する。

| 区分     | 追加する予約語 |
| -------- | -------------- |
| 仮想実行 | `仮に`         |

`仮に` はキーワード完全一致優先（LEX-007）により、識別子分離より先に認識される。

### 3.3 `そうでなければ` の流用

`そうでなければ` はすでに `もし` 構文で予約済みの語であり、意味も「else / それ以外」で一致している。`仮に` でも同じ語を用いることで予約語を増やさず、かつ「仮定が崩れた場合」という読み方を自然に成立させる。

## 意味論

### 4.1 ブロックの役割

`仮に` ブロック（**仮定ブロック**）と `ならば` ブロック（**実行ブロック**）はどちらも**仮想環境**で実行される。

```
仮想環境 = 現在の環境のスナップショット
           + 仮定ブロックで宣言された束縛
```

- 仮定ブロックは `は` 束縛と `変わる` 束縛のみを含む。仮定ブロック自体の評価値は捨てる。
- 実行ブロックは仮想環境上で評価され、最終値を `仮に` 式の値として返す。

### 4.2 副作用の封じ込め（分離規則）

`ならば` ブロック内で行われた `変える`（再束縛）や可変束縛への書き込みは、**外側の環境に反映しない**。ブロック終了時に仮想環境はロールバックされる。

```kotoba
変わる カウンターは 0

結果は 仮に
  カウンターは 99
ならば
  カウンターを カウンターと1の和に 変える
  カウンター

※ 実行後、カウンター は 0 のまま（仮想環境の変更は漏れない）
※ 結果 は 100（仮想環境での最終値）
```

### 4.3 例外ルーティング

`ならば` ブロック実行中に例外が発生した場合：

- `そうでなければ` ブロックが存在する場合: 例外を捕捉し、`そうでなければ` ブロックを**外側の環境**で実行する。`そうでなければ` の値が `仮に` 式の値になる。
- `そうでなければ` ブロックが存在しない場合: 例外はそのまま外側に伝搬する。

この挙動は `試す〜失敗した場合` と対称である。

| 構文          | 成功時の値      | 失敗時の値                       |
| ------------- | --------------- | -------------------------------- |
| `試す` 本体   | 本体の最終値    | `失敗した場合` の値 / `無`       |
| `仮に` ならば | ならば の最終値 | `そうでなければ` の値 / 外へ伝搬 |

### 4.4 戻り値

`仮に` 式は `ならば` ブロックの最終値をそのまま返す（シンプルモデル）。

仮想環境の差分情報が必要な場合は、利用者が `ならば` ブロック末尾で対応表を明示的に組み立てる。

## コード例

### 5.1 基本形（失敗なし）

```kotoba
結果は 仮に
  xは 0
  yは 10
ならば
  xを yで 割る

※ 結果 は 0（割り算の結果）
※ 外側の x, y は存在しないか、あっても変わらない
```

### 5.2 失敗ルーティング

```kotoba
結果は 仮に
  xは 0
ならば
  xを xで 割る         ※ 0÷0 で実行時エラー
そうでなければ
  「その仮定は危険だった」を 返す

※ 例外が発生し、そうでなければ が実行される
```

### 5.3 外側の変数を読んで仮定を上書き

```kotoba
料金は 1000
割引率は 0.1

割引後は 仮に
  料金は 料金と割引率の積
ならば
  料金                  ※ 仮定ブロックで上書きした 料金 を返す

※ 外側の 料金 は 1000 のまま
※ 割引後 は 100.0
```

### 5.4 `試す` との組み合わせ

`仮に` に `そうでなければ` を付けない場合、例外は外に伝搬するので利用者が `試す` で包める。

```kotoba
試す
  結果は 仮に
    xは 0
  ならば
    xを xで 割る
失敗した場合
  「ゼロ除算」を 表示する
```

## 実装方針（VM 参考）

本節は規範でなく、実装者向けの設計ヒントである。

実装パイプラインは `Lex → Parse → Sema → Compile → VM` であり、`check` は VM まで進まない前提がある。`仮に` の仮想実行は **VM 実行時**に実現する。

### VM 側の要点

1. **スナップショット**: `仮に` に入るとき、現在の変数フレームとヒープをスナップショットする。安全優先なら完全コピー、速さ優先なら copy-on-write（書いた時点で初めて複製）。
2. **仮定ブロックの評価**: スナップショット上に仮定束縛を積む。
3. **実行ブロックの評価**: 仮想環境上で実行し最終値を保存する。例外は捕捉してフラグを立てる。
4. **ロールバック**: ブロック終了後、スナップショット前の状態に戻す。
5. **値の返却**: 例外なし → 最終値を返す。例外あり → `そうでなければ` ブロックを外側環境で実行してその値を返す。`そうでなければ` がなければ例外を再送出する。
