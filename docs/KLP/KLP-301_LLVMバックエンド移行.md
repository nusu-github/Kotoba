# KLP-301 — LLVMバックエンドへの移行

> 種別: Kotoba Language Proposal (KLP) — Standards Track（バックエンド移行）  
> 番号: KLP-301  
> 題名: LLVMバックエンドへの移行 — ネイティブコード生成による高性能化  
> 状態: Draft  
> 注記: 本提案は言語仕様が一通り確定した後に着手する。言語の意味論規範（`[規範]` ラベル付き）は含まない。
> 採択後は `docs/実装/設計メモ_v1.md` および `docs/仕様/09_実行モデル.md` に追記する。

---

## 要約

現在の RegVM（レジスタ型バイトコードVM）を廃止し、**LLVM**を用いたネイティブコード生成バックエンドへ移行する。  
これにより言語のパフォーマンスを大幅に向上させ、将来的な最適化・デバッグ情報・ターゲットプラットフォーム対応の基盤を整える。

**移行の前提条件**: 言語仕様（`docs/仕様/`）が一通り確定していること。  
言語の意味論を固定してからバックエンドを切り替えることで、リグレッションの混乱を避ける。

---

## 動機

### RegVM の現状と限界

KLP-300 で確立した RegVM は、言語プロトタイピングと仕様検証の場としては十分に機能する。  
しかし本格的な利用を想定すると、以下の課題が顕在化する。

- **実行速度**: ツリーウォーク／バイトコードVMは解釈オーバーヘッドを免れない
- **最適化の限界**: インライン化・定数畳み込み・デッドコード除去などの機械語レベルの最適化が難しい
- **デバッグ情報**: DWARF など標準デバッグ情報の生成が煩雑
- **ターゲット対応**: x86-64/ARM/WASM など複数アーキテクチャへの対応コストが高い

### LLVMを選ぶ理由

| 観点             | 理由                                                                                |
| ---------------- | ----------------------------------------------------------------------------------- |
| 最適化パス       | LLVM は豊富な最適化パス（`-O2`/`-O3`）を持ち、利用側の実装コストが低い              |
| マルチターゲット | x86-64、ARM64、WebAssembly など主要ターゲットをカバー                               |
| エコシステム     | `inkwell`（Rust バインディング）により Rust から安全に利用可能                      |
| デバッグ情報     | DWARF 生成のサポートが組み込まれており、将来のデバッガ統合（KLP-122）と親和性が高い |
| 実績             | Clang、Rust、Swift など多数の言語実装に採用された信頼性                             |

## 移行方針

### 基本方針

1. **言語仕様確定が先** — 構文・意味論の変更が落ち着いてから移行する
2. **段階的移行** — RegVM と LLVM バックエンドを一時期並走させ、適合テストで同等性を確認
3. **RIRを境界に** — 現在の `RIR`（レジスタIR）を LLVM IR への変換の入力とする。フロントエンドは変更しない
4. **適合テスト通過を基準** — RegVM で通過していた全適合テストが LLVM バックエンドでも通過することを移行完了の条件とする

### 実行パイプライン（移行後）

```
ソースファイル
  ↓ 字句解析（logos）
Token列
  ↓ 文法正規化（chumsky）
正規化Token列
  ↓ 構文解析（Parser）
AST
  ↓ 意味解析（Sema）
TypedHIR（型付き高水準IR）
  ↓ コード生成（Codegen）
RIR（レジスタIR命令列）
  ↓  ← ここから変わる
LLVM IR
  ↓ LLVM 最適化・コンパイル
ネイティブオブジェクトコード / 実行バイナリ
```

### 段階的移行計画

| フェーズ | 内容                                                                       |
| -------- | -------------------------------------------------------------------------- |
| **P1**   | `inkwell` の導入と最小限の LLVM IR 出力確認（Hello World 相当）            |
| **P2**   | RIR → LLVM IR 変換器の実装（算術・変数・制御フロー）                       |
| **P3**   | 手順呼び出し・クロージャ・再帰のサポート                                   |
| **P4**   | 例外処理（`試す`/`もし失敗したら`）のサポート（`landingpad` または別方式） |
| **P5**   | コレクション型（一覧・対応表）とランタイムライブラリの整備                 |
| **P6**   | 適合テスト全通過の確認、RegVM の廃止、デフォルトバックエンドの切り替え     |

## 影響範囲

| 領域                         | 影響内容                                                             |
| ---------------------------- | -------------------------------------------------------------------- |
| `src/backend/vm.rs`          | RegVM 廃止（P6 完了後）                                              |
| `src/backend/codegen.rs`     | LLVM IR 生成器への置き換え（RIR → LLVM IR）                          |
| `src/backend/value.rs`       | Value 型はランタイムライブラリへ移動（C ABI または Rust ランタイム） |
| `src/backend/rir.rs`         | LLVM IR 変換の入力として維持（または統合）                           |
| `Cargo.toml`                 | `inkwell` 依存の追加                                                 |
| `docs/実装/設計メモ_v1.md`   | パイプライン記述の更新                                               |
| `docs/仕様/09_実行モデル.md` | 実行モデル規範は変わらない（意味論は同一）                           |
| `tests/conformance/`         | 適合テストは変更なし（通過を移行基準として使用）                     |
