# KLP-123 — 非同期実行（`背景で`・`待つ`・`しながら`）

> 種別: Kotoba Language Proposal (KLP) — Standards Track（言語機能追加）  
> 番号: KLP-123  
> 題名: 非同期実行 — `背景で`・`待つ`・`しながら` による協調型タスク実行  
> 状態: Draft  
> 注記: 本文書は言語仕様（`docs/仕様/`）への提案である。`[規範]` ラベルは含まず、
>       採択後に対応する仕様ファイルへ規範として移記される。

## 要約

予約済みの3キーワード `背景で`・`待つ`・`しながら` に意味論を与え、
**協調型非同期実行モデル**を言語仕様として確立する。

新型 `タスク<T>` を導入し、`背景で` でタスクを作成、`待つ` で結果を取り出し、
`しながら` で複数タスクを並行待機できるようにする。

現行仕様 RUN-003 は「これらキーワードの評価は常に未実装エラー」と定めているが、
本 KLP の採択をもってその規定を置き換える。

## 動機

### 2.1 IO バウンドの並行処理需要

ファイル読み込み・ネットワーク通信・外部コマンド実行など、CPU ではなく
待ち時間がボトルネックになる処理は多い。現在のことばでは、これらを逐次実行
するしか手段がなく、複数の独立した IO を並行して走らせることができない。

```kotoba
※ 現状：逐次しか書けない
設定は 「config.json」を 読む          ※ 完了まで待つ
テンプレートは 「template.html」を 読む ※ 完了まで待つ
```

非同期機能が入れば、独立した IO を並行実行できる：

```kotoba
設定タスクは 「config.json」を 読むを 背景で 呼ぶ
テンプレートタスクは 「template.html」を 読むを 背景で 呼ぶ
設定とテンプレートは 設定タスクとテンプレートタスクを しながら 待つ
```

### 2.2 「しながら」は日本語の自然な表現

「仕事をしながら音楽を聞く」——日本語において「A をしながら B をする」は
二つの処理を並行して行うことを意味する自然な表現である。
ことばがこの構文を持つことで、並行処理のコードが直感的に読める。

### 2.3 て形チェインとの親和性

ことばの `て形チェイン` は処理の連鎖を表す。非同期処理も連鎖として書ける：

```kotoba
「api/data」に リクエストするを 背景で 呼んで 結果を 待って 表示する
```

### 2.4 予約語は既に存在する

`しながら`・`待つ`・`背景で` はすでに字句定義済みで、パーサーが認識する。
現時点では DGN-006 を発行するだけだが、意味論を定義する準備は整っている。

## 構文

### 3.1 EBNF 差分

```ebnf
(* statement に非同期文を追加 *)
statement ::= ... | background_expr_stmt | await_expr_stmt | concurrent_await_stmt ;

(* 背景実行: 式を タスク<T> にラップして即座に起動 *)
(* 「X を 背景で 呼ぶ」「X を 背景で 実行する」等 *)
background_expr_stmt ::= expr "を" "背景で" verb_stop ;

(* 待機: タスク<T> を T に解決する *)
await_expr_stmt      ::= expr "を" "待つ" ;

(* 並行待機: 複数タスクを同時に待ち、全完了後に値を組として返す *)
concurrent_await_stmt ::= concurrent_task_list "を" "しながら" "待つ" ;
concurrent_task_list  ::= expr ("と" expr)+ ;
```

`background_expr_stmt` および `await_expr_stmt` は式として `postfix_expr` の
一形式にもなる（束縛の右辺や、て形ステップとして使用できる）。

```ebnf
(* 式として埋め込み可能な形式 *)
postfix_expr ::= ... | background_postfix | await_postfix ;

background_postfix ::= access_expr "を" "背景で" verb_stop ;
await_postfix      ::= access_expr "を" "待つ" ;
```

### 3.2 予約語の状態変更

本 KLP の採択により、以下のトークンが「将来予約」から「正式予約語」に昇格する。

| トークン   | 区分変更            | 役割           |
| ---------- | ------------------- | -------------- |
| `背景で`   | 将来予約 → 正式予約 | 背景実行修飾子 |
| `待つ`     | 将来予約 → 正式予約 | 非同期待機動詞 |
| `しながら` | 将来予約 → 正式予約 | 並行待機結合子 |

## 意味論

### 4.1 タスク型（`タスク<T>`）

`タスク<T>` は、将来の任意の時点で `T` 型の値（または例外）を生成する
非同期計算を表すファーストクラス値である。

- `タスク<T>` は **変数に代入でき**、手順の引数・戻り値として渡せる。
- `タスク<T>` は参照カウントまたは所有権モデルで管理する（実装定義）。
- `タスク<T>` が誰にも `待つ` されずにスコープ外に出た場合の動作は未定義とする（OQ-6）。

`タスク<T>` は基本型に追加される新しい組み込み型である。

```
基本型: 数 | 文字列 | 真偽 | 一覧<T> | 対応表<K,V> | 手順 | 無 | タスク<T>
```

### 4.2 背景実行（`背景で`）

```kotoba
タスク は 処理を 背景で 呼ぶ
```

`X を 背景で verb` は次の意味を持つ：

1. `X` を評価する（呼び出し可能な式であること）。
2. ランタイムのタスクキューに `X` の実行を登録し、即座に `タスク<T>` を返す。
3. 実際の処理はイベントループが空いたときに開始される（協調スケジューリング）。
4. `verb` は `呼ぶ`・`実行する` 等の「実行を表す動詞」を使う（OQ-2 参照）。

`背景で 呼ぶ` は `を` 助詞の引数を一つ取る特殊動詞として扱われる。

```kotoba
※ 引数付き手順を背景で呼ぶ
データ取得タスクは データを取得する【対象: 「users」】を 背景で 呼ぶ

※ ラムダ相当を背景で起動する（KLP-TBD と連携）
計算タスクは 重い計算するを 背景で 呼ぶ
```

### 4.3 待機（`待つ`）

```kotoba
結果は タスクを 待つ
```

`タスク を 待つ` は次の意味を持つ：

1. 現在のタスクを**サスペンド**し、対象タスクの完了を待つ。
2. イベントループは他の準備済みタスクを実行し続ける（ブロッキングではない）。
3. 対象タスクが正常完了したとき、その戻り値を返して現在のタスクを再開する。
4. 対象タスクが例外で終了したとき、その例外を現在のタスクで再 raise する。

`待つ` の戻り型: `タスク<T>` に対して `待つ` を適用した結果の型は `T` である。

```kotoba
※ 結果の型は T（タスク<T> → T）
テキストは 読み込みタスクを 待つ  ※ タスク<文字列> → 文字列
カウントは 集計タスクを 待つ      ※ タスク<数> → 数
```

### 4.4 並行待機（`しながら`）

```kotoba
結果Aと結果Bは タスクAとタスクBを しながら 待つ
```

`タスクA と タスクB を しながら 待つ` は次の意味を持つ：

1. 全てのタスクが完了するまで現在のタスクをサスペンドする。
2. イベントループは渡された全タスクを並行してスケジュールする。
3. 全タスク完了後、各タスクの結果を `と` で連結した**組**として返す。
4. いずれかのタスクが例外で終了したとき、その例外を再 raise する（OQ-3）。

戻り型: `タスク<A> と タスク<B> を しながら 待つ` の型は `(A, B)` の組である。

```kotoba
※ 3タスク並行待機
設定とテンプレートとデータは
  設定タスクとテンプレートタスクとデータタスクを しながら 待つ
```

### 4.5 非同期手順の定義

明示的な `非同期` マーカーは**不要**である。任意の手順の内部で
`背景で` および `待つ` を使用できる。

手順を `背景で 呼ぶ` と、その手順全体が `タスク<T>` として起動される。
手順内で `待つ` を使うと、その手順はイベントループと協調するタスクになる。

```kotoba
HTMLを生成する という 手順 【テンプレートURL: を〔文字列〕】
  テンプレートは 「テンプレートURL」を 読むを 背景で 呼ぶ
  データは データAPIから 取得するを 背景で 呼ぶ
  テキストとjsonは テンプレートとデータを しながら 待つ
  テキストを jsonで 展開するを 返す
```

### 4.6 て形チェインとの統合

`await_postfix` は `postfix_expr` の一形式であるため、て形チェインに接続できる。

```kotoba
※ 背景で起動してすぐ待つ（逐次 IO の代替なし、例示目的）
結果は 処理するを 背景で 呼んで 待って 加工して 表示する
```

```kotoba
※ 並行起動してチェインで処理
設定とデータは
  「config.json」を 読むを 背景で 呼ぶと
  「data.json」を 読むを 背景で 呼ぶを
  しながら 待って
  解析して
  表示する
```

### 4.7 エラー処理との統合（`試す`・`必ず行う`）

タスク内で発生した例外は `待つ` を評価した時点で呼び出し元に伝搬する。
`試す` ブロック内で `待つ` を使うと、タスクの例外を捕捉できる。

```kotoba
試す
  結果は 危険な処理するを 背景で 呼んで 待つ
  結果を 表示する
例外 エラー として
  「処理に失敗しました: 」とエラーを 表示する
```

### 4.8 型検査規則

| 規範 ID   | 内容                                                                                                                                                             |
| --------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ASYNC-001 | `X を 背景で 呼ぶ` の `X` は呼び出し可能な型でなければコンパイルエラー                                                                                           |
| ASYNC-002 | `背景で 呼ぶ` の戻り型は `タスク<T>`（`T` は `X` の戻り型）                                                                                                      |
| ASYNC-003 | `T を 待つ` の `T` は `タスク<U>` でなければコンパイルエラー                                                                                                     |
| ASYNC-004 | `タスク<U> を 待つ` の戻り型は `U`                                                                                                                               |
| ASYNC-005 | `しながら 待つ` のオペランドはすべて `タスク<Ti>` でなければコンパイルエラー                                                                                     |
| ASYNC-006 | `しながら 待つ` の戻り型は各 `Ti` を要素とする組                                                                                                                 |
| ASYNC-007 | `タスク<T>` は束縛変数に格納できるが、`一覧<T>` の要素型 `T` として `タスク<T>` を使う場合は `タスク<タスク<T>>` でなく `タスク<T>` として推論する（フラット化） |
| ASYNC-008 | `しながら 待つ` は2つ以上のタスクを必要とする（1つのタスクには `待つ` を使う）                                                                                   |

### 4.9 診断要件

| 診断コード | 規範違反  | メッセージ例（日本語）                                                                     |
| ---------- | --------- | ------------------------------------------------------------------------------------------ |
| DGN-130    | ASYNC-001 | 「`背景で 呼ぶ` には呼び出し可能な式を渡してください: `数` は呼び出せません」              |
| DGN-131    | ASYNC-003 | 「`待つ` には `タスク<T>` 型の値が必要です: `文字列` は待機できません」                    |
| DGN-132    | ASYNC-005 | 「`しながら 待つ` のオペランドに `タスク<T>` 以外の型が含まれています: `数`」              |
| DGN-133    | ASYNC-008 | 「`しながら 待つ` には2つ以上のタスクが必要です。1つのタスクには `待つ` を使ってください」 |
| DGN-134    | —         | 「タスクが完了する前に破棄されました（`待つ` で結果を受け取ってください）」                |

## コード例

### 5.1 並行ファイル読み込み

```kotoba
設定とテンプレートを読む という 手順
  設定タスクは 「config.json」を 読むを 背景で 呼ぶ
  テンプレートタスクは 「template.html」を 読むを 背景で 呼ぶ
  設定とテンプレートは 設定タスクとテンプレートタスクを しながら 待つ
  設定を 表示する
  テンプレートを 表示する

設定とテンプレートを読むを 呼ぶ
```

### 5.2 非同期パイプライン（て形チェインとの統合）

```kotoba
APIエンドポイントの一覧 は
  「/api/users」と「/api/products」と「/api/orders」

全データを取得する という 手順
  タスク一覧は APIエンドポイントの一覧の 各で 繰り返す
    それのXから データを取得するを 背景で 呼ぶを 集める ※ 一覧<タスク<文字列>>

  ※ 並行待機（複数タスクをしながら待つ拡張: OQ-7 参照）
  タスク一覧の 各タスクを しながら 待つを 返す

全データを取得するを 呼んで 各で 繰り返す
  それのXを 解析して 表示する
```

### 5.3 エラーハンドリング付き非同期処理

```kotoba
安全に取得する という 手順 【URL: を〔文字列〕】
  試す
    タスクは URLから データを取得するを 背景で 呼ぶ
    タスクを 待つを 返す
  例外 エラー として
    「取得失敗 (」とURLと「): 」とエラーを 表示する
    無を 返す

結果は 「https://api.example.com/data」を 安全に取得するで 呼ぶ
もし 結果が 無 と 等しくない ならば
  結果を 処理して 表示する
```

## 実装ガイダンス

本節は実装者向けのガイダンスである。仕様の一部ではなく規範的拘束力を持たない。

### 6.1 字句・パーサー（変更小）

`src/frontend/token.rs`・`src/frontend/lexer.rs` は変更不要。
トークン定義（`Shinagara`・`Matsu`・`Haikeide`）は既に存在する。

`src/frontend/parser.rs`:
- `parse_verb()` 内の DGN-006 エラー分岐（`Shinagara | Matsu | Haikeide`）を除去
- `parse_postfix_expr()` に `background_postfix`・`await_postfix` のパースを追加
- `parse_statement()` に `concurrent_await_stmt` のパースを追加

### 6.2 AST ノード追加

`src/frontend/ast.rs` に以下を追加：

```rust
pub enum Expr {
    // ... 既存ノード ...
    Background { callee: Box<Expr>, verb_span: Span },
    Await { task: Box<Expr>, span: Span },
    ConcurrentAwait { tasks: Vec<Expr>, span: Span },
}
```

### 6.3 セマンティクス解析

`src/sema/` に以下の処理を追加：

- `タスク<T>` 型を型システムに登録（`Type::Task(Box<Type>)` 相当）
- `Background` ノード: オペランドが呼び出し可能か検査（ASYNC-001）、戻り型を `Task<T>` に設定
- `Await` ノード: オペランドが `Task<T>` か検査（ASYNC-003）、戻り型を `T` に設定
- `ConcurrentAwait` ノード: 全オペランドが `Task<Ti>` か検査（ASYNC-005/008）、戻り型を組に設定

### 6.4 ランタイム（変更大）

`src/backend/value.rs`:
- `Value::Task(...)` バリアントを追加（既に `// === 将来の非同期対応用予約 ===` コメントあり）

`src/backend/` にイベントループを追加：

- **方式 A（推奨）**: Tokio ランタイムを組み込み、タスクを `tokio::task::spawn` で管理
- **方式 B**: 独自キュー＋`async-std` の軽量ランタイム
- **方式 C**: 実装定義（仕様は実行モデルを規定しない）→ OQ-1 参照
