# Kotoba v1 実装設計メモ

## 構成

- `src/main.rs`: CLI (`run/check/test`)
- `src/lib.rs`: モジュール公開と移行互換 re-export
- `src/frontend/`: Token(logos) / Grammar(chumsky) / Parser / AST
- `src/sema/`: 意味検査・型関連の入口
- `src/module/`: ローダー・解決器
- `src/backend/`: codegen / vm / value / rir / builtins
- `src/diag/`: 診断描画
- `src/common/`: Span / Source

## 現在の実装方針

- パイプライン: Lex -> Parse -> Sema -> Compile -> VM
- `check` は ModuleResolve -> Lex -> Parse -> Sema を実行し、Compile/VM は実行しない。
- `test` は YAML マニフェスト駆動で accept/reject を検証する。
- `run/check` はモジュール解決（相対パス + `.kb`補完 + 循環検出）を先行実行する。
- `test` はマニフェスト品質（重複ID・空入力・`@` プレースホルダ・同一入力過剰重複）を先に検証する。
- ハングアップ暫定対策として、`KOTOBA_PARSE_STEP_LIMIT` / `KOTOBA_ANALYZE_STEP_LIMIT` を有効化している。

## v1 規範への対応要点

- `run/check/test` CLI を提供。
- `これ/それ/あれ` の単独使用を意味検査で拒否。
- 診断は `種別/位置/原因/修正指針` を保持。
- `使う` を `StmtKind::Use` に正規化し、公開定義のみを取り込む。
- `「Xから Yを 使う」` の公開項目検証を実装。

## フロントエンド移行結果（Logos/Chumsky）

- 解析経路は `SourceFile -> Lexer(logos) -> grammar::normalize_token_stream(chumsky) -> Parser` に統一した。
- 識別子・キーワードの語彙正規化は NFC のみを採用した。
- 移行は機能フラグや二重実行を挟まない直接置換で進め、`run/check/test` を同一経路に揃えた。
- 検証は `tests/unit/frontend_snapshot.rs` のスナップショットと `tests/unit/{lexer_prop,parser_prop}.rs` のプロパティテストを基準に運用する。

## 次段階で拡張する点

- TypedHIR / 制約ベース型推論の本体実装。
- RegVM 命令系への完全移行。
- 台帳ケースの品質厳密化（規範別の最小再現入力と重複削減）。
